# 🏗️ 技術選定理由 (Architecture)

本プロダクト（診療材料新規採用申請アプリ）で採用した技術スタックと、選定した理由について記述します。

---

## 1. フロントエンド / ホスティング
* **Next.js (App Router)**
  * **選定理由:** 以前Laravelを触った際、ルーティングファイルに1つずつパスを設定するのが大変でした。Next.jsのApp Routerはフォルダ構造がそのままURLになるため直感的で扱いやすいと感じたことと、Webアプリ開発で最近特によく使われているフレームワークだと聞いたため採用しました。
* **Vercel**
  * **選定理由:** Next.jsの開発元であるため、同じサービスで揃えることで設定の手間や予期せぬトラブルを減らせると考えたためです。実際にCloudflare Pagesにリポジトリをデプロイしたこともありますが、レンダリングの種類によっては設定に工夫が必要と知り、今回のポートフォリオ作成ではトラブル回避と開発スピードを最優先してVercelを採用しました。

## 2. データベース / ORM / 認証
* **Neon (PostgreSQL)**
  * **選定理由:** 過去にレンタルサーバー（ConoHa WING）でMySQLを使用した際、ローカルと本番で接続設定を管理するのが難しく、動作確認のために本番環境へ切り替えた際に誤って本番データを直接操作してしまう危険性を感じました。DBの構築や運用は自力でやらずクラウドに頼る方が安全だと考えています。Supabaseも検討しましたが、GUIでの操作が多くAIに操作方法を聞いても画面の仕様変更などで正しい回答が得られにくく詰まりやすいため、よりシンプルにDB機能へ特化したNeonを選びました。また、GitHub ActionsでPlaywrightを実行する際、毎回Dockerで重いDBコンテナを立てるより、Neonのブランチ機能を使う方が軽量で扱いやすいと知ったため採用しました。
* **Drizzle ORM**
  * **選定理由:** PrismaとDrizzleの比較記事でコードを見比べた際、記号の使い方やコードの並びも含めて、Drizzleの方が自分にとって見慣れた形をしており、スムーズにコードを読めると感じたため採用しました。
* **Clerk**
  * **選定理由:** Supabaseのような多機能サービスで管理画面から複雑な設定をする手間や、自前実装の手間を省くためです。今回のポートフォリオは「QAエンジニアとしてPlaywrightを使ったテスト自動化」に力を入れることが最優先目的のため、認証は単体で分かりやすい専用サービス（Clerk）に頼り、本題のテスト開発に時間を割く判断をしました。

## 3. テスト / 自動化環境
* **Playwright**
  * **選定理由:** Cypressなどの他ツールと比較して、現在のWeb開発で主流になっていると聞いたためです。テスト同士が干渉して失敗しにくい仕組みになっている点や、画面上のマウス操作からテストコードを自動生成してくれる機能（`codegen`）があり、効率よくテスト作成を進められると感じたため採用しました。
* **GitHub Actions**
  * **採用目的:** リポジトリへのプルリクエスト作成などをトリガーにして、PlaywrightのE2Eテストを自動実行し、さらにVercelへのデプロイ連携までを自動化するCI/CD環境として活用しています。
