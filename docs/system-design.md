# 📐 システム・DB設計 (System Design)

システム全体のアーキテクチャ、データベース設計、およびクラス/モジュール分割の判断理由について記述します。

---

## 1. DB設計（ER図・トレードオフの意思決定）

### Q: `users` テーブルのカラム名を `clerk_user_id` とし、`auth_provider_id` 等に抽象化しなかった理由は？

**A: 抽象化によるオーバーエンジニアリングを避け、将来的な認証プロバイダ移行時の安全性を担保するためです。**

具体的な理由は以下の2点です。

1. **抽象化の費用対効果（ROI）が低い:**
   認証サービス（IdP）の変更は頻繁に起こるものではなく、発生確率が低いリスクに対して初めから抽象的なデータ構造を組むメリットが少ないと判断しました。
2. **移行時の安全性が高い（マルチカラム保持）:**
   もし将来 Clerk から別の IdP（例: Supabase Auth や Auth0）に移行する場合、`clerk_user_id` とは別に `supabase_user_id` などの新しいカラムを一時的に併設する方が、旧IDとのマッピングやデータ移行（マイグレーション）作業を段階的かつ安全に行えます。
